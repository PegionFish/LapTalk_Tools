# `csv_visual` 多 CSV 对比能力实施方案

## 1. 背景

`csv_visual` 当前已经具备以下能力：

- 加载单个 HWiNFO CSV 文件
- 选择多个参数叠加绘图
- 对单个文件的参数配置独立颜色
- 在后台自动生成预览，并导出透明 PNG

现有实现的核心前提是“单文件状态模型”：

- GUI 只维护一个 `self.data`
- 参数选择使用单个 `selected_column_indices`
- 参数颜色使用按列索引存储的 `column_colors`
- 裁剪范围、预览请求、导出逻辑都围绕单个 `HWiNFOData` 组织

这意味着当前实现无法直接扩展为多文件对比，主要原因包括：

1. 不同 CSV 中相同列索引会冲突
2. 不同 CSV 的录制起止时间不同，需要人为对齐
3. 当前裁剪范围只针对单个文件，而不是针对“对齐后的比较视图”
4. 当前图例标签无法清晰区分“同名参数来自哪个文件”

本方案的目标是在**不破坏单文件工作流**的前提下，为 `csv_visual` 增加多 CSV 对比能力，并把“手动对齐”作为一等能力来设计。

---

## 2. 目标与范围

### 2.1 本次功能目标

本次改造需要完整覆盖以下需求：

1. 多个 CSV 文件的导入、移除
2. 不同文件的时间轴手动对齐
3. 不同文件、不同参数的独立颜色配置
4. 自动预览与导出在多文件模式下继续可用
5. 单文件模式保持与现有行为一致

### 2.2 明确纳入范围

- 支持一次添加多个 CSV
- 支持拖拽多个 CSV
- 支持移除单个或多个已加载 CSV
- 支持设置“基准文件”
- 支持对每个文件输入正负时间偏移量
- 支持跨文件选择参数
- 支持给跨文件参数单独配色
- 支持对齐后的全局时间范围裁剪
- 支持多文件图例区分
- 支持 README 和测试同步更新

### 2.3 明确不纳入首版范围

以下能力在首版中不做，以控制实现复杂度：

- 自动波形匹配对齐
- 根据峰值/阈值自动推断偏移量
- 将多文件配置持久化到磁盘
- 记忆上一次会话的文件列表与颜色配置
- 提供多图层布局或子图对比模式
- 对数据进行插值、重采样或补齐

首版只做“手动对齐、叠加绘制、可视化对比”。

## 3. 设计原则

### 3.1 保留现有单文件核心能力

以下现有能力不应被推翻重写，而应在其上方增加一层“多文件会话模型”：

- `HWiNFOData`
- `load_hwinfo_csv(...)`
- `extract_series(...)`
- `preload_numeric_series(...)`
- 当前 `ChartStyle` 样式体系

### 3.2 手动对齐必须是核心交互，不是附属选项

因为不同 CSV 的开始/结束点可能完全不同，所以：

- 不能假设所有文件从 `00:00:00` 开始就已经对齐
- 不能假设文件结束点相同
- 不能把“对齐”设计为导出前最后一步，而必须在预览中持续可见

### 3.3 不修改原始数据，只修改绘图视图

对齐应该作用于“渲染时使用的时间轴”，而不是写回 CSV 数据本身。

即：

- 原始 `elapsed_seconds` 保持不变
- 实际绘图使用 `aligned_elapsed_seconds = elapsed_seconds + offset_seconds`

### 3.4 用跨文件唯一键替换裸列索引

现有颜色和选择状态都基于单个 `column_index`，这在多文件下会冲突。

首版必须把“参数身份”升级为：

- `SeriesKey = (session_id, column_index)`

而不是继续使用：

- `column_index`

## 4. 核心数据模型

## 4.1 新增 `LoadedCsvSession`

建议新增一个会话级数据结构，用来描述“一个已加载的 CSV 文件”。

建议字段：

```python
@dataclass
class LoadedCsvSession:
    session_id: str
    alias: str
    data: HWiNFOData
    offset_seconds: float = 0.0
    is_reference: bool = False
    is_visible: bool = True
    preload_ready: bool = False
    preload_error: str | None = None
```

字段说明：

- `session_id`：内部唯一标识，避免路径改名或同名文件冲突
- `alias`：用户在界面中看到的文件别名，默认取文件名 stem
- `data`：对应的 `HWiNFOData`
- `offset_seconds`：对齐偏移量，可正可负
- `is_reference`：是否为当前基准文件
- `is_visible`：是否参与绘图，首版可先固定为 `True`
- `preload_ready` / `preload_error`：后台预载状态

## 4.2 新增 `SeriesKey`

建议新增不可变键，用于唯一标识“某个文件中的某个参数”：

```python
@dataclass(frozen=True)
class SeriesKey:
    session_id: str
    column_index: int
```

用途包括：

- 参数选中状态
- 参数颜色映射
- 图例标签构建
- 预览请求与渲染请求

## 4.3 新增 `SeriesDescriptor`

为了减少 GUI 层反复拼接显示文本，建议增加一个描述对象：

```python
@dataclass(frozen=True)
class SeriesDescriptor:
    key: SeriesKey
    session_alias: str
    column_display_name: str
    legend_label: str
```

其中：

- `column_display_name`：例如 `[002] CPU Package Power [W]`
- `legend_label`：例如 `RunA · [002] CPU Package Power [W]`

## 5. 时间轴对齐设计

## 5.1 首版对齐模型

每个文件有一条原生相对时间轴：

- `raw_elapsed_seconds`

每个文件再有一个手动输入的偏移量：

- `offset_seconds`

最终绘图时间轴：

- `aligned_elapsed_seconds = raw_elapsed_seconds + offset_seconds`

语义如下：

- `offset_seconds > 0`：曲线整体向右移动
- `offset_seconds < 0`：曲线整体向左移动
- `offset_seconds = 0`：不移动

## 5.2 为什么必须允许负时间

当某个文件要向左移动时，对齐后时间轴前段可能出现负值。

例如：

- 基准文件从 `0s` 开始
- 第二个文件需要左移 `8s`

那么第二个文件的前几个点会出现在 `-8s ~ 0s` 区间。

因此首版建议：

- 允许全局时间轴出现负值
- 滑块与标签也允许显示负时间

不要在内部偷偷把所有曲线再次整体平移到非负区间，否则用户理解成本会上升。

## 5.3 基准文件规则

基准文件用于帮助用户理解“谁是 0 点参照物”。

推荐规则：

- 首个导入文件默认设为基准文件
- 基准文件的 `offset_seconds` 初始为 `0`
- 其他文件相对基准文件调整

## 5.4 切换基准文件的处理规则

切换基准文件时，不能让现有可视布局跳变。

推荐做法：

1. 找到新基准文件当前的 `offset_seconds`
2. 将所有文件的 `offset_seconds` 都减去这个值
3. 将新基准文件标记为 `is_reference = True`

这样可以保证：

- 新基准文件的偏移量变成 `0`
- 所有曲线相对位置保持不变
- 用户视觉上不会看到整体图像跳动

## 5.5 对齐后的全局可视范围

首版裁剪范围必须从“单文件裁剪”升级为“对齐后的全局裁剪”。

全局边界定义为：

- `global_start = min(session.offset_seconds)`
- `global_end = max(last_elapsed + session.offset_seconds)`

其中 `last_elapsed` 为该文件原始相对时间轴的最后一个点。

裁剪逻辑应当作用于这个全局区间，而不是作用于某个单文件。

## 5.6 首版不做的对齐增强

首版不做：

- 对齐参考线
- 自动对齐建议
- 峰值标记吸附
- 鼠标拖拽对齐

但要给后续增强留接口。

## 6. GUI 方案

## 6.1 总体布局调整

现有顶部“单个 CSV 文件输入框”需要升级为“多文件管理区”。

推荐布局：

1. 顶部：文件管理区
2. 左侧：参数过滤与参数列表
3. 左侧中下：图表样式与裁剪
4. 左侧下方：参数颜色
5. 右侧：预览
6. 底部：状态栏

## 6.2 文件管理区

推荐新增一个 `Labelframe`，包含：

- 文件列表（建议用 `ttk.Treeview`）
- 操作按钮：
  - `添加 CSV...`
  - `移除选中`
  - `清空全部`
  - `设为基准`

文件列表建议列：

- `别名`
- `文件名`
- `时长`
- `偏移`
- `基准`

推荐原因：

- 文件列表需要多列信息
- 需要快速识别哪个文件是基准
- 需要快速查看当前偏移值

## 6.3 文件详情编辑区

在文件列表下方增加一个简洁的编辑区，用于编辑当前选中的单个文件：

- `别名` 输入框
- `偏移(秒)` 输入框
- 微调按钮：
  - `-10s`
  - `-1s`
  - `+1s`
  - `+10s`

原则：

- 文件列表负责“看全局”
- 文件详情区负责“精细编辑”

## 6.4 参数列表区

参数列表仍建议使用“平铺列表 + 过滤”，不要在首版换成树状多层控件。

列表项显示格式建议为：

- `[RunA] [002] CPU Package Power [W]`
- `[RunB] [003] GPU Core Load [%]`

过滤规则建议同时匹配：

- 文件别名
- 原始列名
- 显示名

例如用户输入：

- `RunA`
- `CPU`
- `GPU load`

都应能筛出对应项。

## 6.5 已选参数颜色区

现有“参数颜色”区保留，但内容升级为跨文件参数列表。

列表项显示格式建议为：

- `[RunA] [002] CPU Package Power [W] · #66CCFF`

行为保持与现有一致：

- 支持手输 HEX
- 支持取色器
- 支持清除颜色

唯一变化是：

- 颜色映射对象从 `column_index` 改为 `SeriesKey`

## 6.6 对齐后裁剪控件

现有“起始裁剪 / 结束裁剪”滑块保留，但含义改为：

- 针对“对齐后的全局时间轴”

变更点：

1. 滑块范围可能出现负值
2. 标签要支持显示负时间
3. 裁剪值不是基于单个 `self.data`
4. 文件偏移改变时，滑块最大/最小边界要自动刷新

## 7. 交互流程

## 7.1 导入流程

### 浏览导入

新增：

- `askopenfilenames(...)`

代替当前单文件 `askopenfilename(...)`。

导入流程：

1. 用户点击“添加 CSV...”
2. 可一次选择多个文件
3. 对每个文件执行解析
4. 成功的文件加入会话列表
5. 失败的文件弹汇总错误
6. 成功文件立即可参与筛选和预览
7. 数值序列仍在后台预载

### 拖拽导入

当前拖拽逻辑只提取第一个 CSV，需要改为提取所有 CSV。

新规则：

- 拖入多个路径时，筛出所有 `.csv/.CSV`
- 保持用户拖入顺序
- 去重后依次导入

## 7.2 去重规则

首版建议按“规范化绝对路径”去重。

规则：

- 如果同一路径已加载，再次导入时不重复创建会话
- 状态栏提示“已存在，已跳过”
- 可将已有项聚焦到文件列表中

这样可以避免：

- 用户重复拖入同一文件造成混乱
- 颜色和对齐状态被复制

## 7.3 移除流程

用户点击“移除选中”后：

1. 从已加载会话中删除对应 `LoadedCsvSession`
2. 删除该文件相关的参数选择状态
3. 删除该文件相关的颜色配置
4. 清理对应的后台预载状态
5. 如果移除的是基准文件，则自动选择首个剩余文件为新基准并重归一化偏移
6. 刷新参数列表、颜色列表、预览与裁剪范围

## 7.4 空状态

当没有已加载文件时：

- 参数列表为空
- 参数颜色列表为空
- 预览恢复占位提示
- 状态栏显示“请添加一个或多个 HWiNFO CSV 文件”

## 8. 渲染层方案

## 8.1 保留现有 `build_figure(...)`

现有单文件渲染函数应保留，避免：

- 单文件测试大面积失效
- 未来需要单文件导出时丢失简单路径

## 8.2 新增多文件渲染函数

建议新增：

```python
def build_comparison_figure(
    sessions: Sequence[LoadedCsvSession],
    selected_series: Sequence[SeriesKey],
    width_px: int = 1920,
    height_px: int = 1080,
    dpi: int = 160,
    style: ChartStyle | None = None,
    color_by_series: Mapping[SeriesKey, str] | None = None,
    visible_range_seconds: tuple[float, float] | None = None,
) -> Figure:
    ...
```

## 8.3 绘图规则

对每个 `SeriesKey`：

1. 找到对应 `LoadedCsvSession`
2. 从 `session.data.extract_series(column_index)` 获取原始序列
3. 将 X 轴全部加上 `session.offset_seconds`
4. 按全局可视范围裁剪
5. 组装图例标签为 `alias · display_name`
6. 根据 `color_by_series` 决定颜色
7. 绘制到同一个坐标轴中

## 8.4 图例与标签规则

图例文本必须区分文件来源。

建议格式：

- `RunA · [002] CPU Package Power [W]`
- `RunB · [002] CPU Package Power [W]`

这样即使参数名相同，也不会混淆。

## 8.5 多文件模式下的空数据判断

以下情况都应明确报错：

- 没有选中任何参数
- 选中参数在当前可视范围内全部没有数据点
- 所有文件都为空或被移除

错误提示要保持现有风格，避免静默失败。

## 8.6 输出文件名

建议新增多文件输出名辅助函数，例如：

```python
build_comparison_output_name(...)
```

推荐输出名模式：

- `compare__RunA__RunB__cpu_gpu.png`

若文件过多，可缩写为：

- `compare__3_files__cpu_gpu.png`

原则：

- 文件名可读
- 不过长
- 与单文件输出名规则并存

## 9. 预览与后台线程方案

## 9.1 预览请求对象升级

当前预览请求只支持单个 `data`，需要升级为多文件版本。

建议新增：

```python
@dataclass(frozen=True)
class ComparisonPreviewRenderRequest:
    request_id: int
    sessions: tuple[LoadedCsvSession, ...]
    selected_series: tuple[SeriesKey, ...]
    width_px: int
    height_px: int
    dpi: int
    style: ChartStyle
    color_by_series: dict[SeriesKey, str]
    visible_range_seconds: tuple[float, float] | None
```

## 9.2 预载线程策略

当前预载线程一次只处理一个 `HWiNFOData`，首版可以继续沿用“单个请求、逐个执行”的简单策略。

建议：

- 每添加一个文件，提交一个预载请求
- 结果返回时按 `session_id` 更新 `preload_ready / preload_error`
- 不需要为了首版引入复杂线程池

## 9.3 预览刷新策略

以下操作都应触发预览刷新：

- 添加文件
- 移除文件
- 设置基准文件
- 修改偏移量
- 选择/取消选择参数
- 修改颜色
- 修改图表样式
- 修改全局裁剪范围

与当前保持一致：

- 普通输入防抖刷新
- 明确操作可立即刷新

## 10. 纯函数抽取建议

为了避免关键逻辑都塞进 GUI，建议把以下逻辑抽成纯函数，便于单元测试：

1. `build_series_descriptors(...)`
2. `compute_global_time_bounds(...)`
3. `align_series_x_values(...)`
4. `normalize_offsets_for_reference(...)`
5. `filter_visible_series(...)`
6. `build_comparison_output_name(...)`

其中尤其重要的是：

- `normalize_offsets_for_reference(...)`
- `compute_global_time_bounds(...)`

这两个函数与“手动对齐”体验直接相关，应优先做成可测试的纯逻辑。

## 13. 测试方案

## 13.1 单元测试

## 13.2 GUI 测试

1. 添加 2 个文件后参数列表包含两组参数
2. 移除其中 1 个文件后，相关选中项和颜色一起消失
3. 修改偏移值后，预览请求携带更新后的偏移
4. 多文件模式下导出仍使用当前样式和颜色

## 13.3 手工验证

5. 为不同文件的曲线指定不同颜色
6. 导出透明 PNG
7. 检查导出结果与预览一致

## 14. 风险点与处理策略

## 14.1 风险：GUI 改造面较大

问题：

- 当前 GUI 假设只有一个文件

处理：

- 先抽纯逻辑，再改 GUI
- 优先复用现有参数列表、预览、样式配置区

## 14.2 风险：负时间显示会影响现有格式化逻辑

问题：

- 当前时间格式化默认只处理非负值

处理：

- 新增带符号格式化函数
- 单文件模式仍走现有格式

## 14.3 风险：预览请求对象体积变大

问题：

- 多文件请求会带多个 `HWiNFOData`

处理：

- 首版可接受
- 后续若有性能问题，再考虑快照精简

## 14.4 风险：用户对“基准文件”和“偏移方向”理解不一致

问题：

- 正负偏移语义容易混淆

处理：

- 在 GUI 文案中明确说明：
  - 正值向右移
  - 负值向左移
- 给出微调按钮，减少纯心算成本
